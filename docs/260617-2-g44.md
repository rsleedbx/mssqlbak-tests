## 1. Physical structure

The G44 large-string columnstore dictionary consists of **two LOB blobs** stored in the
same `_collect_blobs` map:

| Blob ID | Version | entry_count | Size (deint.) | Role |
|---------|---------|-------------|---------------|------|
| 2001 | 4 | **1199** | 79,204 bytes | Hash index — Huffman-encoded string pool |
| 23844 | 7 | 0 (field unused) | 28,003 bytes | Sorted string pool — 194 bookmark entries |

Blob 2001 is identified by `version=4` at byte offset 0 of the raw LOB.  
Blob 23844 is identified by `version=7` at byte offset 0 of the raw LOB and the sentinel
`entry_size == 103` at the start of the bookmark block.

Both blobs are wrapped in the columnstore LOB chunk format (12-byte preamble + 8-byte
separators between 65536-byte data chunks). `_deinterleave_column_lob` strips these to
produce a flat logical stream.

## 2. Blob 2001 — v4 hash-dict layout

Offsets are into the **deinterleaved** byte stream.

| Offset | Size | Field | Value (this fixture) |
|--------|------|-------|----------------------|
| 0 | 4 | `version` | 4 |
| 4 | 4 | `entry_count` | 1199 — strings in the Huffman bitstream |
| 8 | 4 | `hash_slots` | 8192 |
| 12 | 4 | `page_count` | 2 — number of Huffman pages |
| 16–43 | 28 | (zeros / reserved) | — |
| 44 | 4 | (unknown) | 1 |
| 48 | 4 | (unknown) | 8 |
| 52 | 4 | `total_distinct_count` | 1200 — full dictionary size including blob-23844 strings |
| 56–67 | 12 | (unknown) | 1, 4, 1 |
| 68 | 1199 × 8 | Record handles: `(bit_offset: u32, page_id: u32)` each LE | — |
| 9660 | 40 | Page index (see §2.1) | — |
| 9700 | 128 | `encode_array` — nibble-packed canonical Huffman code lengths for 256 symbols | — |
| 9832 | declared 69,380 bytes | Compressed bitstream (`CBUF`) | — |

`entry_count = 1199`: there are exactly 1199 record handles (k=0..1198).
`total_distinct_count = 1200`: the full dictionary has 1200 entries; the 1200th string
(data_id=7) is stored in blob 23844, not in the Huffman bitstream.

### 2.1 Page index (offsets 9660–9699)

The 40 bytes between the last record handle and `encode_array` form a page index. The
fields are partially decoded:

| Offset | Value | Interpretation |
|--------|-------|----------------|
| 9660 | 554567 | `bit_offset` of the k=1199 phantom handle (see §5 note) |
| 9664 | 0 | `page_id` of phantom handle |
| 9668 | 69540 | (unknown — not CBUF byte length; may be total raw bitstream bytes including separator and page header) |
| 9672 | 6 | (unknown) |
| 9676 | 0 | (reserved) |
| 9680 | 1200 | `total_distinct_count` (repeated from header offset 52) |
| 9684 | 2 | `page_count` (repeated from header offset 12) |
| 9688 | 555022 | `store_total_bits` (STB) — total bits across all Huffman pages |
| 9692 | 7 | (unknown) |
| 9696 | 69380 | `declared_cbuf_bytes` — declared Huffman bitstream byte length |

`declared_cbuf_bytes = 69380` vs actual deinterleaved CBUF bytes = 69372. The
difference is exactly 8, matching the size of one LOB separator. See §4.2.

### 2.2 LOB separator embedded in the bitstream

The raw blob has the structure:

```
raw[0..11]        — 12-byte LOB preamble (stripped by _deinterleave_column_lob)
raw[12..65547]    — chunk 0: 65536 bytes of data
raw[65548..65555] — 8-byte LOB separator (stripped by _deinterleave_column_lob)
raw[65556..79223] — chunk 1: 13668 bytes of data
```

`CBUF_OFF = 9832`, so within the raw blob, the CBUF starts at `raw[12+9832] = raw[9844]`.
The LOB separator at `raw[65548..65555]` falls at **CBUF byte 65548−9844 = 55704**
(absolute bit 78656 + 55704×8 = **524288**).

Page 1's empirically determined start is bit **524490**, which is 202 bits after the
separator ends (524288 + 64 = 524352). This places the separator inside the page
boundary region between Page 0 and Page 1.

The `declared_cbuf_bytes = 69380` exactly matches the raw CBUF span including the
separator: `79224 − 9844 = 69380`. The current `_deinterleave_column_lob` strips the
separator, leaving 69372 bytes (8 short). This is the root cause of data_id=7's string
extending beyond the deinterleaved buffer boundary.

## 3. Huffman codec

### encode_array

Located at `deint2001[9700:9828]`. Each nibble encodes the Huffman code length for one
of the 256 byte-value symbols. Nibble `i` at nibble-index `n` covers symbol `n`.

To build the canonical Huffman table:
1. Expand nibble-packed array to 256 `(symbol, length)` pairs.
2. Filter `length > 0` to get the active symbols.
3. Sort by `(length, symbol)`.
4. Assign canonical codes in that order (increment code, shift left on length increase).
5. Fill a flat lookup table: for each code, extend to `max_len` bits and map all
   extensions to `(symbol, length)`.

`max_len = 7` for this fixture. The Huffman alphabet covers symbols 50–124.

### Character mapping

`char = chr(symbol - 2)`.  
The 80-character strings use the set `[0-9A-Za-z]` plus "XX" suffix.

### Pair-swap

The compressed buffer uses the xVelocity pair-swap transformation: adjacent bytes are
swapped before Huffman decoding.  Apply `pair_swap` to `deint2001[CBUF_OFF:]` before
decoding. In practice it is simpler to pair-swap the full `deint2001` (the non-CBUF
region decodes identically with or without swapping because record-handle reads are
unaffected).

### Buffer padding

Pad `deint2001` with at least 8 zero bytes **before** the pair-swap to avoid a 4-byte
lookahead overrun when decoding strings near the very end of the bitstream.

## 4. Two-page structure

Despite all record handles having `page_id = 0`, the bitstream is logically split
into two pages.

### Page 0 (k=0..962)

Formula for string k's start bit:

```
start_bit(k) = CBUF_bits + bit_offset[k] + 5
```

where `CBUF_bits = 9832 × 8 = 78656` and `bit_offset[k]` is the 32-bit LE value at
`deint2001[68 + k×8]`.

The constant 5 (SHIFT) was determined empirically.

### Page 1 (k=963..1198)

The formula above produces the **wrong** starting positions for k ≥ 963.
For k=963, the formula gives bit 524087; the actual start is bit **524490** (gap = 403
bits). The gap does not follow the handle's `bit_offset`.

Chain-decode strategy:
1. Seek to the empirically determined Page 1 start: **bit 524490**.
2. Decode string k=963 forward; its end bit is `pos_end`.
3. For k=964..1198: `start_bit = pos_end + 5` (5-bit inter-string padding).
4. If primary decode fails, retry with `start_bit ± delta` for `delta` in a small
   range (±12 bits covers accumulated drift in this fixture).

Page 1 verified: 236/237 handles successfully chain-decoded, 0 mismatches against
`G44.json` ground truth.

## 5. The data_id=7 gap

`entry_count = 1199` means blob 2001 contains exactly 1199 strings. The full dictionary
has 1200 entries. The missing string is **data_id=7** ("HAwmq66vRZJ9BUwO...XX").

It is **not** in blob 2001's bitstream. The wide-scan diagnostic (624000..633632 bits)
found data_id=994 as the last valid string in blob 2001 (ending at bit 633614);
everything after that decodes to zero-padding artifacts.

data_id=7 is stored in blob 23844's 194-entry bookmark block.

## 6. Blob 23844 — v7 sorted pool

### Physical layout (raw, not deinterleaved)

| Offset | Content |
|--------|---------|
| 0 | `version = 7` (u32 LE) |
| 4 | (unused / entry_count field reads 0) |
| 1945 | Start of 194-entry bookmark block |

Each bookmark entry is 103 bytes:

| Offset within entry | Size | Field |
|--------------------|------|-------|
| 0 | 2 | `entry_size` = 103 (u16 LE sentinel) |
| 2 | 80 | 80-char ASCII string |
| 82 | 8 | Unknown metadata |
| 90 | 4 | `step_minus_1` (f32 LE) |
| 94 | 9 | Unknown trailing bytes |

`step_minus_1` encodes `step − 1` where `step` = number of data_ids until the next
bookmark (i.e., the bookmark's rank in the sorted dictionary increments by `step`
between consecutive entries).

### Current decoder (`_parse_v7_sorted_pool`)

Reads all 194 bookmark entries and returns a list of 1200 elements where the 194
bookmark ranks contain their string and all other positions are `None`.

The decoder outputs strings indexed by their **sorted rank**, not by data_id.

### Missing piece: rank → data_id attribution

The blob 23844 decoder knows the sorted rank of each bookmark string but not its
data_id. The Huffman decoder (blob 2001) knows data_ids for k=0..1198 from the record
handles.

To attribute data_id=7 to its blob-23844 bookmark:

1. Collect all 1199 strings decoded from blob 2001 → `{data_id: string}`.
2. Sort that dictionary by string value → produces the sorted rank for each data_id.
3. The blob-23844 bookmark at the rank where data_id=7's string would fall is
   data_id=7's entry.

Alternatively: in `_parse_v7_sorted_pool`, cross-reference each bookmark string
against the strings already decoded from blob 2001. The one bookmark string not
present in blob 2001's output is data_id=7.

## 8. Implementation summary

`_decode_v4_huff_dict(raw, deint)` in `columnstore.py`:

1. Reads `n = deint[4:8] (u32) + 1` (entry count from last data-id).
2. Reads `n` record handles (bit offsets) from `deint[68 : 68 + n*8]`.
3. Reads `encode_array` from `deint[9700:9828]` and `stb` from `deint[9688:9692]`.
4. Passes `raw[9844:]` (raw CBUF, includes LOB separator) to `xmhuffman.decode_page`.
5. Applies `chr(b - 2)[1:]` to each decoded `bytes` result (symbol offset + overhead strip).
6. Returns `sorted(decoded)` — alphabetical rank = data_id.

Key: using `raw[9844:]` (not the deinterleaved slice) preserves the 8-byte LOB separator
at chunk boundary so xmhuffman's internal page-skip aligns correctly.

## 10. Hints from the 4-degree sweep (2026-06-17)

4-degree search from SQLEspresso resolved the two highest-priority G44 unknowns:

| Missing piece | Source | Resolution |
|--------------|--------|------------|
| **Page boundary / LOB separator** | `XMHUFFMAN-PYPI` (PyPI v0.3.0) | **Resolved**: `decode_page` uses `end = offsets[s+1] if s+1 < n else total_bits`; separator bytes fall in an inter-string gap and decode harmlessly; no explicit page-header parsing needed |
| **Page 1 base derivation** | `XMHUFFMAN-PYPI` | **Resolved**: no explicit Page 1 base needed; the library uses handle offsets directly and `total_bits` as end sentinel |
| **entry_count=1199 vs 1200 handles** | blob header + `XMHUFFMAN-PYPI` | **Resolved**: 1200 record handles (k=0..1199) cover all 1200 strings; `entry_count=1199` is hash-table entry count, not string count |
| **Hash function** | `D0NK-VPAQ`, `PBIXRAY-DICT` | Still unknown; non-blocking since direct k indexing works |

**SQLEspresso Part 2** corroborates the separator-within-bitstream finding: columnstore
LOB data is stored via SQL Server’s native LOB storage engine ("22486 lob read-aheads"),
which does not align the Huffman bitstream to LOB chunk boundaries.

**Implementation complete.** `_decode_v4_huff_dict` in `columnstore.py` performs this
call and is wired into both enc=3 row-read paths.  All 1200/1200 strings verified against
`G44.json` by `test_g44_large_dict_full_coverage` (requires `xmhuffman` ≤ Python 3.11).
