## 1. Summary

Six bugs have been confirmed in the CCI segment decoder (`mssqlbak/columnstore.py`).  They
cluster into five distinct root-cause classes:

| Bug ID | Class | Types affected | Symptom | Root cause |
|--------|-------|---------------|---------|------------|
| **E3A** | Wrong `as_bytes` flag | `decimal_38_10`, `datetimeoffset_7`, `uniqueidentifier` | Binary dict entries decoded as Latin-1 text; values return garbage or `''` | `_parse_dict_strings` called with `as_bytes=False` for binary-payload types |
| **E3B** | No offset table for fixed-width binary | `binary_8`, `binary(n)` | All non-null rows return `None` | `_parse_dict_strings` looks for a variable-length offset table; `BINARY(n)` has none |
| **E3C** | Compact-RLE null vector mis-parse | `nvarchar_50` | NULL rows return `''` instead of `None` | NULL flag cleared incorrectly; row gets dict lookup at index 0 (empty string) |
| **K3A** | Char dictionary index mis-read | `char(n)` with Ōēź26 distinct values | Structural rows return `''` instead of the expected string | Dict index or entry-length error in the enc=3 CHAR path with large dictionaries |
| **K3B** | Null bit-vector polarity inversion | `varbinary(n)`, `bit` with sparse-NULL | All non-null rows decode as `None`; the one actual NULL is not affected | Null mask polarity inverted for sparse-NULL segments (few NULLs, mostly non-null) |
| **K3C** | UUID decoder crash | `uniqueidentifier` with Ōēź1,200 dict entries | `ValueError: bytes_le is not a 16-char string` | E3A produces Latin-1 strings of variable length; some are <16 bytes, crashing `uuid.UUID(bytes_le=...)` |

### Resolution status (2026-06-19)

| Bug | State | Evidence |
|-----|-------|----------|
| **E3A** | Ō£ģ Fixed | `_parse_dict_strings` now uses `as_bytes=True` + typed decode for binary-payload types; `test_enc3_bugs.py` E3A tests pass on all 4 versions |
| **E3B** | Ō£ģ Fixed | fixed-width BINARY pool now decoded; `binary_8`/`binary_16` return bytes |
| **E3B-v2** | Ō£ģ Fixed | `BINARY(4)` no longer reads the pool header as an offset table (`cci_binary4`) |
| **E3C** | Ō£ģ Fixed | NVARCHAR null row decodes as `None` |
| **K3B** | Ō£ģ Fixed (Format C var-len) | enc=5 Format C variable-length VARBINARY/BINARY, commit `b9f78af` |
| **K3C** | Ō£ģ Fixed | `cci_uuid` enc=5 Format C UNIQUEIDENTIFIER now routed away from the ARCHIVE heuristic; `test_cci_types_large_coverage.py` cci_uuid passes on all 4 versions |
| **K3A** | Ō£ģ Fixed | enc=3 v4 Huffman dictionary now returned in native `data_id` order, not sorted alphabetically; VARCHAR(50)/NVARCHAR(50) structural rows decode correctly on all 4 versions |

**Correction (K3C root cause):** the earlier claim that K3C was the E3A `as_bytes` bug
and would be fixed automatically is **disproven**.  The `cci_uuid` segment is enc=5
**Format C** (`h92=0`, `u16@38 == n_rows`), not enc=3.  It crashed because the
Format-C-priority pre-check in `_decode_enc5` only covered `BINARY`/`VARBINARY`, so
`UNIQUEIDENTIFIER` fell through to the ARCHIVE type-2 heuristic.  That heuristic
misfired ŌĆö a `0xFFFF` byte in the XPRESS stream at offset 236, a coincidental `u32 ==
n_rows`, and a 16-divisible implied pool all matched ŌĆö producing variable-width garbage
(1,151/1,200 `None`, the rest 1ŌĆō232 bytes) that crashed `uuid.UUID(bytes_le=ŌĆ”)`.  Adding
`UNIQUEIDENTIFIER` to the Format-C-priority set decodes 1,199 valid 16-byte UUIDs + 1
NULL.  Fix: `mssqlbak/columnstore.py` `_decode_enc5`.

**K3A root cause (resolved):** the enc=3 dictionary for these columns is an xVelocity
**v4 Huffman** dictionary decoded by `_decode_v4_huff_dict`.  That function decoded each
entry by its record-handle index `k` (the `data_id` that column segments store) but then
**`sorted()` the values alphabetically before returning**, discarding the `k` order.
Column stored-index `s` resolves to `dictionary[s ŌłÆ 2]`, so after sorting, index 0 pointed
at the alphabetically-first string instead of the first `data_id`.  This was invisible
whenever insertion order happened to equal alphabetical order (e.g. `cci_char` with
low/high `'A'`/`'B'` ahead of AŌĆōZ fillers ŌĆö which is also why a stale generator edit had
"corrected" `cci_char` to `'A'`/`'B'` to match the *buggy* output).  `cci_varchar50`
(low `'L'`, high `'Z'`) and `cci_nvarchar50_sparse` break that coincidence and exposed the
bug.  Fix: return `[decoded_by_k[k] for k in range(n)]` ŌĆö native `data_id` order, no sort.
The `cci_char` generator constants were reverted to `'L'`/`'H'` to match the committed
`.bak` (verified against the generator at the commit that built it, `ee70691`).

All seven bugs (E3A, E3B, E3C, E3B-v2, K3A, K3B, K3C) are fixed; no `xfail` markers remain
for this register.

## 2. Why the 4-row fixture hid these bugs

SQL Server selects encoding based on the statistics of the segment.  For a 4-row segment
(created by `REORGANIZE WITH (COMPRESS_ALL_ROW_GROUPS = ON)` on a 4-row table) there are
at most 4 distinct values per column.  With so few distinct values, SQL Server always uses
**enc=5** (raw bytes, one value per row).  The dictionary encoder (enc=3) is never invoked.

The K-1 fixture added 1,196 NULL filler rows (total 1,200).  The K-3 fixture added 1,197
**non-null** varied filler rows.  In both cases, with 1,200 rows, SQL Server selects enc=3
for string and binary columns.  The different null-to-non-null ratio between K-1 and K-3
changes the null bit-vector format, which exposed K3A and K3B.

## 4. Bug E3A ŌĆö wrong `as_bytes` flag for binary-payload dictionary types

### Affected types

`decimal_38_10`, `datetimeoffset_7`, `uniqueidentifier`

### Manifestation

| Type | Expected decoded value | Actually decoded |
|------|----------------------|-----------------|
| `decimal_38_10` low | `Decimal("1.0000000000")` | Multi-byte Latin-1 garbage string (raw 17-byte decimal blob) |
| `datetimeoffset_7` low | `datetime(2000, 1, 1, 0, 0, tzinfo=utc)` | Multi-byte Latin-1 garbage string (raw 8-byte binary) |
| `uniqueidentifier` low | `UUID("00000000-ŌĆ”")` | `''` (zero UUID binary decoded as zero-length text) |

### Root cause

`_parse_dict_strings` accepts an `as_bytes` flag that controls whether dictionary entries are
returned as `bytes` or decoded as text.  For `decimal_38_10`, `datetimeoffset_7`, and
`uniqueidentifier`, the flag is set to `False` (text mode).

The 17-byte decimal binary, 8-byte datetimeoffset binary, and 16-byte UUID binary are all
decoded via `bytes.decode('latin-1')`, producing Latin-1 garbage strings instead of the raw
bytes that the type-specific decoders (`_scaled_decimal`, `_decode_datetimeoffset`,
`_decode_uuid`) expect.

For `uniqueidentifier` specifically: the 16 zero bytes of the low UUID decode as a 16-character
Latin-1 string consisting of null characters (`\x00` ├Ś 16).  The UUID dispatcher treats this
as a malformed value and returns `''`.

### Fix location

`mssqlbak/columnstore.py` ŌĆö the call-site(s) that invoke `_parse_dict_strings` for each SQL
type.  The `as_bytes` argument must be `True` for all types whose dictionary entries are
raw binary payloads:

- `DECIMAL` / `NUMERIC`
- `DATETIMEOFFSET`
- `UNIQUEIDENTIFIER`

## 5. Bug E3B ŌĆö fixed-length BINARY dictionary returns empty list

### Affected types

`binary_8` (K-1), `binary(16)` (K-3 `cci_binary`)

### Manifestation

| Type | Row | Expected | Actually decoded |
|------|-----|----------|-----------------|
| `binary_8` low | id=1 | `b'\x00\x00\x00\x00\x00\x00\x00\x00'` | `None` |
| `binary_8` high | id=2 | `b'\xff' * 8` | `None` |
| `binary_8` mid | id=3 | `b'\x80' + b'\x00' * 7` | `None` |
| `binary(16)` low | id=1 | `b'\x00' * 15 + b'\x01'` | `None` |
| `binary(16)` high | id=2 | `b'\xff' * 16` | `None` |

All non-null rows decode as `None` because the dictionary list is empty.

### Root cause

`_parse_dict_strings` is written for variable-length types: it reads an **offset table** at
the start of the dictionary blob (a sequence of uint16 or uint32 offsets, one per entry) and
uses those offsets to slice out entries.

`BINARY(n)` is **fixed-width**: its dictionary blob contains `N ├Ś width` raw bytes laid
consecutively with **no offset table prefix**.  `_parse_dict_strings` walks the absent offset
table and finds no valid entries, returning an empty list `[]`.  When `_decode_enc3` looks up
index `i` into `[]` it gets `None`.

### Fix location

`mssqlbak/columnstore.py` ŌĆö add a `_parse_dict_fixed(blob, entry_width)` helper (or a
`fixed_width` branch inside `_parse_dict_strings`).  For fixed-width binary types, slice the
blob into `entry_width`-byte chunks sequentially:

```python
def _parse_dict_fixed(blob: bytes, entry_width: int) -> list[bytes]:
    return [blob[i:i+entry_width] for i in range(0, len(blob), entry_width)]
```

The type width must be threaded from the column metadata (available in the segment header,
already parsed into the `ColumnSegment` object) into the dictionary parser call-site.

### Tests that prove the bug exists (xfail, will xpass when fixed)

```

tests/test_enc3_bugs.py::test_binary8_high_value_is_bytes
```

### Tests that must pass after the fix (currently xfail ŌåÆ will pass)

```
tests/test_tabletype_cci_large_coverage.py::test_cci_low[binary_8]
tests/test_tabletype_cci_large_coverage.py::test_cci_high[binary_8]
```

## 6. Bug E3C ŌĆö compact RLE null vector confuses NULL with empty string

### Affected types

`nvarchar_50`

### Manifestation

| Row | Expected | Actually decoded |
|-----|----------|-----------------|
| `null` (id=4) | `None` | `''` (empty string) |
| `low` (id=1) | `''` (empty string ŌĆö valid low value for nvarchar) | correct |
| `high` (id=2) | `'z' * 50` | correct |

Non-null rows decode correctly.  Only the null-sentinel row is affected.

### Root cause

The compact RLE encoding for the null bit-vector (used when `has_null=True` in the segment
header) encodes runs of null and non-null flags with (flag, count) pairs.  There is an
off-by-one or sign error in the RLE decoder: the null flag for the null-sentinel row is
decoded as "non-null".  Because the null flag is cleared, the decoder proceeds to a dictionary
lookup.  Dictionary index 0 for `nvarchar_50` is the empty string `''`, so that is returned.

Note: Bug E3C is distinct from Bug K3B (┬¦8).  E3C affects a specific row under the dense-NULL
(many NULLs in K-1) compact-RLE format.  K3B affects **all non-null rows** under the
sparse-NULL (few NULLs in K-3) format.  They are likely separate code paths in the null
bit-vector decoder.

### Fix location

`mssqlbak/columnstore.py` ŌåÆ `_decode_enc3` null-bitmap application, or the compact RLE
decoder that precedes it.  The null flag for a given row position must be checked **before**
any dictionary lookup; if the flag is set, return `None` immediately.

### Tests that prove the bug exists (xfail, will xpass when fixed)

```

```

### Tests that must pass after the fix (currently xfail ŌåÆ will pass)

## 7. Bug K3A ŌĆö CHAR(20) wrong value with large dictionary (Ōēź26 entries)

### Affected types

`char(n)` in a compressed segment with Ōēź26 distinct values

### Manifestation

| Row | Expected | Actually decoded |
|-----|----------|-----------------|
| low (id=1, `'LLLLLLLLLLLLLLLLLLL'`) | `'LLLLLLLLLLLLLLLLLLLL'` (20 L's) | `''` (empty string) |
| high (id=2, `'HHHHHHHHHHHHHHHHHHHH'`) | `'HHHHHHHHHHHHHHHHHHHH'` | `''` (empty string) |
| null (id=3) | `None` | `None` Ō£ģ |
| filler rows (idŌēź7, AŌĆōZ cycling) | 20-char strings | correct Ō£ģ |

Structural (manually inserted) rows return `''`; filler rows (generated by tally-table INSERT)
decode correctly.  The bug is therefore not universal ŌĆö only specific dictionary entries or
row positions are affected.

### Context

K-1's `char_10` column has only 3 distinct non-null values (low, high, mid) + 1,197 NULLs.
With 4 total dictionary entries the structural rows decode correctly.

K-3's `cci_char` has 26 distinct filler values (AŌĆōZ cycling) + 2 structural values (L├Ś20,
H├Ś20) = **28 dictionary entries**.  The structural rows are inserted first (ids 1, 2, 3) and
the filler rows second (ids 7ŌĆō1,203).  After REORGANIZE, segment row ordering may differ
from insertion order.  The structural rows specifically return `''`.

### Likely root cause

Two candidate explanations:

1. **Dictionary index overflow / modulo error**: The dictionary index stored for the
   structural rows points to an entry beyond the 26-entry filler sub-range.  A modulo or
   truncation in the index decoder wraps it to 0 (the empty-string entry) for large
   dictionaries.

2. **CHAR fixed-stride vs. NCHAR/VARCHAR stride confusion**: The `_parse_dict_strings` code
   for CHAR may use a fixed stride of 10 bytes (inherited from the `char_10` case) rather
   than 20 bytes, mis-slicing entries in the dictionary pool.  With stride 10 on a
   CHAR(20) pool, entries would be half-sized and the structural rows (which have specific
   20-char values) would map to wrong sub-slices.

### Fix location

`mssqlbak/columnstore.py` ŌĆö the CHAR(n) branch of `_parse_dict_strings` or `_decode_enc3`.
Ensure the stride/width used to slice dictionary entries matches the column's declared `n`
(available from the segment header), not a hard-coded constant.  Also verify that dictionary
index decoding correctly handles indexes Ōēź26.


## 8. Bug K3B ŌĆö null bit-vector polarity inversion for sparse-NULL segments

### Affected types

`varbinary(n)` and `bit` in a compressed segment where NULLs are rare (Ōēż1 in 1,200 rows)

### Manifestation

All rows ŌĆö including rows with concrete non-null values ŌĆö are decoded as `None`.  The one
actual NULL row is also returned as `None` (which happens to be correct, but for the wrong
reason).

| Fixture | Expected non-null rows | Actually decoded |
|---------|------------------------|-----------------|
| `cci_varbinary` low (b'\x01') | `b'\x01'` | `None` |
| `cci_varbinary` high (b'\xff'├Ś16) | `b'\xff' * 16` | `None` |
| `cci_varbinary` filler (idŌēź7) | bytes objects | `None` (all 1,197 rows) |
| `cci_bit` low (False) | `False` | `None` |
| `cci_bit` high (True) | `True` | `None` |
| `cci_bit` filler (alternating) | True / False | `None` (all 1,197 rows) |

Row counts are correct (1,200 rows found per table), confirming the rows themselves are
parsed ŌĆö only the `val` column is entirely nulled out.

### Contrast with K-1 (mostly-NULL)

In K-1, `bit` works correctly.  The key difference:

| Fixture | Non-null rows | NULL rows | Null vector character |
|---------|--------------|-----------|----------------------|
| K-1 `bit` | 3 | 1,197 | Dense-NULL: long run of NULLs ŌåÆ compact RLE or run-length block |
| K-3 `cci_bit` | 1,199 | 1 | Sparse-NULL: nearly all non-null ŌåÆ single-bit or inverted format |

The two null vector formats are likely decoded by the same code path using a polarity
convention that is correct for dense-NULL but inverted for sparse-NULL.

### Likely root cause

The null bit-vector blob uses **0 = null, 1 = non-null** (or vice versa) depending on
a format flag in the segment header.  The decoder either:

- Ignores this format flag and always treats the bit as one polarity, or
- Correctly reads the flag but applies it with the wrong boolean sense.

In the sparse-NULL case (K-3), the format flag likely signals "inverted" polarity (where
`1` means null, to allow run-length compression of the non-null run).  The decoder does
not invert, so all non-null bits (`1`) are read as null.

### Fix location

`mssqlbak/columnstore.py` ŌĆö the null bit-vector reader used by `_decode_enc3` (and
potentially `_decode_enc2` for `cci_bit`).  Find where the null mask byte or flag is read
from the segment header and ensure the polarity of the bit (0=null vs 1=null) is correctly
applied.  The fix must handle both dense-NULL (K-1 style) and sparse-NULL (K-3 style)
without regression.

### Tests that must pass after the fix (currently xfail ŌåÆ will pass)

## 9. Bug K3C ŌĆö UNIQUEIDENTIFIER decoder crash in large dictionary

### Affected types

`uniqueidentifier` in a compressed segment with Ōēź1,200 dictionary entries

### Manifestation

The decoder raises an exception before any row is returned:

```
ValueError: bytes_le is not a 16-char string
  File "mssqlbak/types.py", line 388, in _decode_uuid
      return uuid.UUID(bytes_le=raw)
```

The crash occurs during `_decode_enc5` (ARCHIVE path) or `_decode_enc3` (dictionary path)
when a dictionary entry that was supposed to be 16 bytes is shorter.

### Root cause

**K3C is a downstream consequence of Bug E3A.**

E3A causes `_parse_dict_strings` to decode the 16-byte UUID dictionary entries as Latin-1
text (`bytes.decode('latin-1')`).  In a small dictionary (K-1: 3ŌĆō4 entries), the specific
UUID values used happen to round-trip through Latin-1 decode+encode without shortening.  In
a large dictionary (K-3: up to 1,200 entries from 1,197 filler rows), at least one of the
UUID values produces a Latin-1 string shorter than 16 characters (e.g., if the UUID binary
contains `\x00` bytes that get dropped or folded during Latin-1 decode).  When `_decode_uuid`
then tries `uuid.UUID(bytes_le=<short_string>.encode(...))`, it receives fewer than 16 bytes
and panics.

### Fix location

**Same as E3A**: set `as_bytes=True` when calling `_parse_dict_strings` for
`UNIQUEIDENTIFIER`.  Once the dictionary entries are returned as raw `bytes` objects (not
Latin-1 strings), the 16-byte invariant is guaranteed and `_decode_uuid` will not crash.

No separate fix is needed for K3C.

### Tests that prove the bug exists (xfail, will xpass when fixed)

```

tests/test_cci_types_large_coverage.py::test_structural_rows[cci_uuid]
```

### Tests that must pass after the fix (i.e., after E3A is fixed)

