### Issue 1: `enc=1` ARCHIVE integer bitpack — **FIXED** (2026-06-16)

Root cause found and fixed: `_read_large_root_data` was reading LOB fragment records
with `frag[_LOB_DATA:]` (end of record) instead of `frag[_LOB_DATA:bsz]` (bsz-capped).

be included in the assembled blob.  With the wrong blob, `_unwrap_archive_blob` read
the second ARCHIVE chunk header from the wrong offset, producing garbled bitpack data.

Fix: `_read_large_root_data` now caps each fragment at `bsz`.  All 4 SS versions
produce correct sequential id values.  Tests: `test_archive_null_non_null_values`,
`test_archive_null_code_null_positions` — both pass.

## enc=5 CHAR off-by-one: confirmed root cause (2026-06-16)

**Ground truth:** `archive_columnstore_types_full.bak.stats.json` (SS2022) records exactly **70 nulls** for
`archive_char10.val`, confirmed by SQL Server itself on restore.  The decoder returns **69**.  This is a
decoder bug, not a fixture issue.

**Diagnostic finding (`archive_char10`, SS2022, `_decode_enc5_archive`):**

```
n_chunks=7, last chunk (k=6): n_block=2181, pool_len=2180, n_raw=2161, n_valid=2161, n_overflow=20
rows covered: 32820..35000
overflow rows (highest 20, descending): 35000, 34999, …, 34981
chunk 0 pre_meta (global header) first 20 u16s: [1, 0, 1, 0, 0, 0, 10, 0, 7, 0, 69, 0, 512, 0, 2, 0, 8, 0, 0, 35000]
overflow decoded from global header: all 20 entries → pool[0]="32820" or pool[1]="32821" (WRONG)
nulls in overflow: expected=1 (row 35000), got=0  →  lost 1 null
```

**Two co-operating bugs:**

**Bug A — ASCII pool boundary detection overshoots the last chunk.**
- Expected pool: 2181 rows − 5 nulls = **2176 entries**.
- Scan finds `pool_len = 2180` (4 entries too many) because the first few bytes of the
  index section happen to be printable ASCII.
- `index_start` is displaced 40 bytes right → n_raw = 2161 instead of 2181.
- The 20 highest-row-number index entries (rows 34,981–35,000) are silently skipped.

**Bug B — Circular overflow model reads the global header.**
- The 20 skipped entries inflate `n_overflow = n_block − n_valid = 20`.
- The circular model reads these 20 "overflow" entries from `chunk 0's pre_meta`.
- But chunk 0's `pre_meta` is the **global segment header**, not overflow data.
  Its bytes (`[1,0,1,0,0,0,10,0,7,0,69,0,…]`) produce `v=0 → pool[0]` and `v=10 → pool[1]`.
- Row 35,000 (true value = NULL = 0xFFFE) is decoded as `pool[0] = "32820     "`.

**Total impact on `archive_char10` last chunk:**
- All 20 rows 34,981–35,000 have wrong values (non-null rows get pool[0]/pool[1];
  null at row 35,000 becomes non-null).
- Null count test detects only 1 of the 20 wrong rows (the one that should be NULL).

**Fix (IMPLEMENTED 2026-06-16 — TODO-I1 complete):**
- `index_start_k = len(chunk_k) − n_valid_k × 2`
- `pool_len_k = (index_start_k − pool_start_k) // col_width`
- For `k = N-2`: two-pass overflow formula (see below) determines `n_overflow` without relying on ASCII scan.

Both Bug A and Bug B are eliminated by the new formula.

## enc=5 two-pass overflow formula (implemented 2026-06-16)

The `n_overflow` for chunk `k = N-2` (second-to-last) cannot use the raw `pm_lens[N-1]`
because the last chunk's `pre_meta` can contain both chunk `N-2`'s overflow entries **and**
chunk `N-1`'s self-overflow entries ("contaminated" case).

**Algorithm (now in `_decode_enc5_archive`):**

1. **Standard formula**: try `n_ov_std = (pm_lens[N-1] − 6) // 2` for chunk `N-2`.
2. **Validate**: compute `pool_bytes_std = index_start_std − pool_start_{N-2}`.
   Check `pool_bytes_std % col_width == 0`.
3. **If valid** (no contamination): use `n_ov_{N-2} = n_ov_std`, `n_ov_{N-1} = 0`.
4. **If invalid** (contaminated — pool straddles boundary): fall back to **trend extrapolation**:
   - `inc = pm_lens[N-2] − pm_lens[N-3]`
   - `pred = pm_lens[N-2] + inc`
   - `n_ov_{N-2} = (pred − 6) // 2`
   - `n_ov_{N-1} = (pm_lens[N-1] − pred) // 2`

**Why the standard formula can fail:**  When `pm_len` increments are non-constant (e.g.,
alternating 6/8 bytes), the trend extrapolation undershoots `n_ov` by 1, causing a
partial pool entry to straddle the pool/index boundary.  The divisibility check on
`pool_bytes` detects this and falls back to the standard formula.

**Bug C (`archive_null.code` satellite blocks) — FIXED 2026-06-16**

## enc=5 inner blob structure (discovered)

The enc=5 ARCHIVE inner blob (after `_unwrap_archive_blob`) is split into **65536-byte
logical chunks**. Each chunk has this layout:

```
[pre_meta bytes] [0xFFFF u16] [n_block u32] [pool entries] [index u16…]
```

Key findings from the new `archive_columnstore_types_full.bak` investigation:

- **Chunk 0's `pre_meta` is a global segment header**, not an overflow section.
  It contains structured fields including col_width, n_chunks, null_count, n_rows.
  Example (as u16 LE pairs for char10, n_rows=35000, n_chunks=7, null_count=69,
  col_width=10):
  ```
  [1, 0, 1, 0, 0, 0, col_width=10, 0, n_chunks=7, 0, null_count=69, 0,
   512, 0, 2, 0, 8, 0, 0, n_rows=35000, …]
  ```

- **Chunks 1…N**: `pre_meta` contains the overflow u16 entries from chunk k−1,
  followed by a 3-entry sentinel **`[9, 0, 0]`** (6 bytes) at the very end.
  - n_overflow_k = `(len(pre_meta_{k+1}) − 6) // 2`
  - n_valid_k = n_block_k − n_overflow_k
  - index_start_k = `len(chunk_k) − n_valid_k × 2`
  - pool_len_k = `(index_start_k − pool_start_k) // col_width`

- **The last chunk's overflow**: has no circular path back to chunk 0 (chunk 0's
  pre_meta is the global header, not an overflow store). The last chunk likely has
  n_overflow=0 or its overflow count is encoded in the global header.

- The current code assumes a **circular overflow model** (chunk N → chunk 0's
  pre_meta). This is incorrect: the OLD decoder happens to work for CHAR because
  the global header bytes accidentally include values (`0`, `10`, etc.) that pass
  as valid pool offsets, and the count-limit in `_read_overflow` naturally stops
  before the invalid entries.

**Root cause fixed — non-persisted computed columns phantom-assigned storage (commit `f627548`):**

`catalog.py` builds a secondary `rscols_hb` index keyed by `hbcolid` to handle legacy SQL Server 2000 upgraded databases where `rscolid` is a large partition-based key. In modern tables with non-persisted computed columns (e.g. `HumanResources.Employee.OrganizationLevel AS OrganizationNode.GetLevel()`), `hbcolid` is the *physical slot index*, not the logical `colid`. This coincidentally equalled the computed column's logical `colid`, causing the next real variable-length column's storage (a NVARCHAR slot) to be assigned to the computed column. The decoder then read NVARCHAR bytes as a SMALLINT, producing hundreds-of-bits Python integers that overflowed Arrow's int64 with `OverflowError: Python int too large to convert to C long`.

Fix: gate `rscols_hb` population on `rscolid > 4096` — legacy partition keys are always >> any sequential column id; modern small rscolids that differ from hbcolid by one (due to a computed-column gap) are now correctly excluded.

**Impact:** Unblocked all 8 AdventureWorks OLTP databases (2012–2025) and AW2016_EXT.

#### TODO-F1 — Fix single-chunk enc=5 decoder — **FIXED (2026-06-17)**

**Root cause:** ARCHIVE enc=5 blobs for small row groups (n ≤ 5000) are stored in
**type-2 single-chunk format**: no outer XPRESS wrapper (`cmp_sz = 0`), inner data
laid out directly with the standard `[pre_meta][0xFFFF][n_block u32][pool][index]`
structure.  The existing ARCHIVE detectors required either `unc_sz == 65 536` (Path 1,
multi-chunk XPRESS wrapper) or `h38_u16 != n` (Path 2, large n disambiguation);
neither matched type-2 blobs, so they fell through to `_decode_enc5_compressed` →
None → `[None] * n_rows`.

**Fix (2026-06-17):** `mssqlbak/columnstore.py` — added ARCHIVE type-2 detection in
`_decode_enc5`: when `h38_u32 == n` but `h38_u16 == n` (ambiguous for small n), scan
the raw blob for the `0xFFFF` chunk delimiter at a u16-aligned offset ≥ 96 bytes,
followed by `n_block == n` and a non-empty pool region.  When found, route to
`_decode_enc5_archive` instead of the Format C path.

Also added `cmp_sz <= len(blob) - 12` (relaxed from `<`) in Path 1 to catch any
future single-chunk type-1 blobs (cmp_sz == len-12, unc_sz == 65 536).

### The unlock: `DBCC CSINDEX` — the columnstore analogue of `DBCC PAGE`

Source: **Paul White** (Microsoft Data Platform MVP, query-processor internals),
[sql.kiwi — Columnstore Grouped Aggregate Pushdown](https://www.sql.kiwi/2019/04/grouped-aggregate-pushdown/);
syntax originally surfaced by **Niko Neugebauer**, CISL
[part 21 "DBCC CSIndex"](https://www.nikoport.com/2013/11/07/clustered-columnstore-indexes-part-21-dbcc-csindex/).
Undocumented, present **since SQL Server 2012**, test-system only.

```sql
DBCC TRACEON (3604);
DBCC CSINDEX (
    {'dbname' | db_id},
    rowsetid,     -- hobt_id / partition_id from sys.column_store_segments
    column_id,    -- NOTE: +1 for CLUSTERED columnstore, not for nonclustered
    rowgroupid,   -- segment_id
    object_type,  -- 1 = Segment, 2 = Dictionary
    print_option  -- 0 | 1 | 2
    [, start_bitpack_unit] [, end_bitpack_unit]
);
```

**Why this matters:** it exposes exactly what `sys.column_store_segments` hides
and what every open columnstore TODO is blocked on. The output sections (field
names per Paul White's dump):

| Section | Fields | Resolves |
|---------|--------|----------|
| **Segment Attributes** | `Version, encodingType, hasNulls, BaseId, Magnitude, PrimaryDictId, SecondaryDictId, MinDataId, MaxDataId, NullValue, OnDiskSize, RowCount` | **G3** — the encoded min/max + the BaseId/Magnitude (value-encoding/FOR) transform to verify min/max decode |
| **RLE Header + RLE Data** | `Lob type, RLE Array Count (Native Units), RLE Array Entry Size`; per-entry `Index, Bitpack Array Index, Value, Count` | **G6** — the true pure/RLE vs impure split and the null/value layout for large segments |
| **Bitpack Data Header** | `Bitpack Entry Size (= b width), Bitpack Unit Count, Bitpack MinId, Bitpack DataSize` | **G6 / enc=1** — the actual `b` and per-segment base, independent of our guesses |
| **Bitpack Data** | per-unit decoded values (`start`/`end` unit window) | byte-for-byte decoder cross-check |

This is the **columnstore A1 verifier** the architecture doc §13.5 called for, but
better than the DMV: it gives RLE + bitpack + bookmarks. Capturing
`DBCC CSINDEX` sidecars for the failing fixtures is the prerequisite for G6 and G3.

Confirmed format facts from the same source (corroborate, do not replace, our §7):
- RLE fires only at **≥ 64 contiguous** equal values; everything else is bit-packed.
- Bit-pack widths are constrained to those that tile a 64-bit unit on hard borders
  (e.g. 9, 12, 16, 21 — never 11 or 14) — matches verifier-doc §7.2.
- `BaseId` + `Magnitude` are the value-encoding base/scale (FOR) — matches §14.1.
